# AI 小说转剧本工具：Agent 开发题目要求

## 1. 题目名称

**AI 小说转剧本工具**

---

## 2. 题目背景

很多小说作者希望将自己的小说作品改编成剧本，但传统剧本改编通常需要较高的专业门槛，包括：

- 小说叙事与剧本叙事结构不同；
- 小说文本需要拆解为场景、人物、对白、动作、旁白等剧本元素；
- 多章节内容需要保持人物、情节、时间线和场景的一致性；
- 作者需要一个可编辑、可继续打磨的剧本初稿，而不是一次性不可控的生成结果。

因此，本题目要求开发一款 **AI 辅助剧本创作工具**，帮助小说作者降低改编门槛，提升剧本初稿生成效率。

---

## 3. 开发目标

开发一个能够将小说文本自动转换为结构化剧本的 AI Agent / 工具系统。

该工具应支持将 **3 个章节以上的小说文本** 自动转换为 **结构化剧本**，并以 **YAML 格式** 输出，方便作者后续编辑、修改和二次创作。

---

## 4. 核心功能要求

### 4.1 小说文本输入

工具需要支持输入不少于 3 个章节的小说文本。

建议支持以下输入方式之一或多种：

- 直接粘贴小说文本；
- 上传 `.txt` / `.md` 文件；
- 按章节分别输入；
- 从指定目录读取多个章节文件。

---

### 4.2 自动章节解析

工具应能够识别小说中的章节结构，并将输入文本拆分为多个章节。

建议支持：

- 自动识别章节标题；
- 章节顺序保持一致；
- 对章节内容进行分段处理；
- 对超长章节进行合理切分。

---

### 4.3 小说内容理解

AI Agent 应对小说文本进行结构化理解，包括但不限于：

- 主要人物识别；
- 人物关系识别；
- 场景地点识别；
- 时间线识别；
- 关键事件提取；
- 情节冲突提取；
- 对白与叙述性文字区分；
- 可改编为剧本动作的叙述内容提取。

---

### 4.4 自动转换为剧本

工具需要将小说内容转换为剧本初稿。

剧本内容应包含：

- 剧本标题；
- 章节或场次结构；
- 场景信息；
- 时间信息；
- 地点信息；
- 出场人物；
- 人物对白；
- 人物动作；
- 场景描述；
- 旁白或画外音；
- 情节摘要；
- 可供作者编辑的备注信息。

---

### 4.5 YAML 格式输出

最终输出必须是 **结构化 YAML 格式**。

YAML 应具备以下特点：

- 层级清晰；
- 字段命名稳定；
- 可读性强；
- 便于后续程序解析；
- 支持作者手动编辑；
- 能表达剧本中的场景、人物、对白、动作和备注等内容。

---

### 4.6 可编辑剧本初稿

生成结果不要求达到最终成片剧本水平，但需要达到“可继续打磨”的初稿质量。

重点是：

- 帮助作者快速获得结构化剧本草稿；
- 保留小说中的主要情节和人物关系；
- 将小说叙述转化为适合影视 / 短剧 / 广播剧等形式的剧本表达；
- 允许作者后续继续增删场景、修改对白、调整节奏。

---

## 5. 额外文档要求：YAML Schema 说明文档

除工具本身外，必须额外提交一篇文档，用于定义剧本的 **YAML Schema**。

该文档需要包含以下内容：

### 5.1 Schema 字段定义

需要说明 YAML 中每个字段的含义，例如：

- `title`：剧本标题；
- `source`：原始小说信息；
- `characters`：人物列表；
- `chapters`：原小说章节；
- `scenes`：剧本场景；
- `dialogues`：对白；
- `actions`：动作描写；
- `narration`：旁白；
- `metadata`：元信息；
- `notes`：作者或 AI 备注。

---

### 5.2 字段类型说明

需要说明每个字段的数据类型，例如：

- 字符串；
- 数组；
- 对象；
- 布尔值；
- 枚举值；
- 可选字段；
- 必填字段。

---

### 5.3 YAML 示例

文档中应提供一个完整或简化的 YAML 示例，让评委能够理解最终输出结构。

示例可以包含：

```yaml
title: "示例剧本"
source:
  type: "novel"
  chapters_count: 3

characters:
  - id: "char_001"
    name: "林舟"
    role: "protagonist"
    description: "故事主角，性格冷静。"

scenes:
  - id: "scene_001"
    chapter_ref: "chapter_001"
    title: "雨夜相遇"
    location: "旧城区巷口"
    time: "夜晚"
    characters:
      - "char_001"
    summary: "林舟在雨夜遇见神秘女子。"
    beats:
      - type: "action"
        content: "林舟撑着伞，快步穿过狭窄的巷子。"
      - type: "dialogue"
        character: "char_001"
        content: "你是谁？为什么会在这里？"
      - type: "narration"
        content: "雨声遮住了远处传来的脚步声。"
```

---

### 5.4 Schema 设计原因

文档中必须说明为什么这样设计 Schema。

建议从以下角度说明：

#### 1. 面向作者编辑

YAML 相比纯文本更结构化，但仍然容易阅读和手动修改，适合小说作者快速编辑。

#### 2. 面向程序解析

稳定的字段结构便于后续接入：

- 前端可视化编辑器；
- 剧本导出工具；
- 分镜生成工具；
- 人物关系图；
- 剧情时间线；
- 多轮 AI 修改。

#### 3. 保留小说来源信息

通过 `chapter_ref`、`source` 等字段，可以追踪每个剧本场景来自哪一章小说，方便作者对照原文修改。

#### 4. 支持剧本创作流程

将剧本拆分为 `scenes`、`beats`、`dialogue`、`action`、`narration` 等结构，可以更符合剧本创作和影视化改编流程。

#### 5. 支持后续扩展

Schema 应允许未来扩展，例如：

- 分镜字段；
- 镜头语言；
- 情绪标签；
- 场景时长；
- 人物弧光；
- 剧集结构；
- 多版本剧本管理。

## 7. 作品有效性要求

### 7.1 方向一致性

作品必须符合所选议题方向，即围绕 **AI 小说转剧本工具** 进行开发。

不能偏离为其他无关方向，例如：

- 单纯小说生成器；
- 单纯聊天机器人；
- 普通文本摘要工具；
- 不输出剧本结构的文本改写工具。

---

### 7.2 自主完成

作品必须确保自主完成，不得抄袭。

如使用第三方库、框架或模型能力，需要在 README 中明确说明。

---

### 7.3 持续交付要求

开发周期内必须保持持续的 PR 和 commit 记录。

严禁临近截止时间一次性提交所有代码。

以下情况可能被视为无效作品：

- 仅在最后一天一次性导入全部代码；
- commit 时间戳不在所选批次的开始与截止时间之间；
- PR 描述为空；
- PR 描述与实际代码变更严重不符；
- 引用了第三方库或框架，但未在 README 中列明依赖；
- 使用了第三方框架但未说明原创功能部分；
- 复用了自己过去的代码片段，但未在 PR 描述中注明来源。

---

## 8. PR 提交规范

### 8.1 基于 PR 添加新功能

项目开发应通过 PR 推进，而不是直接在主分支上混乱提交。

---

### 8.2 每个 PR 只做一件事

每个 PR 应只实现或修改一个单一功能。

建议：

- 功能粒度尽可能小；
- PR 尽可能清晰；
- 大功能拆分为多个独立 PR；
- 每个 PR 合并后，主分支仍保持可运行。

---

### 8.3 PR 标题要求

PR 标题需要用一句话说明本次新增或修改了什么。

示例：

```text
feat: add chapter parser for multi-chapter novel input
```

或中文：

```text
feat: 新增多章节小说解析模块
```

---

### 8.4 PR 描述要求

每个 PR 描述必须清晰完整，建议包含以下内容：

#### 1. 功能描述

说明该功能的作用和使用方式。

#### 2. 实现思路

简要说明：

- 技术选型；
- 核心实现逻辑；
- 主要模块设计；
- 与其他模块的关系。

#### 3. 测试方式

说明如何验证该功能正常运行，例如：

- 执行了哪些命令；
- 使用了哪些测试文本；
- 预期输出是什么；
- 是否通过单元测试或手动测试。

---

### 8.5 PR 合并后的要求

每个 PR 合并后，主分支代码应保持可运行状态。

评委在任意时间查看仓库时，应能够根据 README 复现演示效果。

总分由以下三部分组成：

---

- 代码是否健壮；
- 逻辑是否清楚；
- 命名是否规范；
- 代码是否可读；
- PR 数量是否合理；
- PR 质量是否较高；

- README 是否完整；
- 依赖说明是否清楚；

重点考察：

- Demo 视频是否清晰；
- 是否完整展示核心功能；
